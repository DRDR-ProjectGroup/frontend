## 🐛 문제 원인

```txt
Next.js App Router의 prefetch 동작으로 인해,
<Link>가 뷰포트에 노출되면 해당 페이지의 서버 컴포넌트가 미리 실행된다.

이 과정에서 페이지 내부의 글 상세 조회 API가 호출되고,
백엔드에서 해당 API 호출을 조회수 증가 조건으로 사용하고 있어
사용자가 실제로 페이지에 진입하지 않았음에도 조회수가 증가하는 문제가 발생한다.
```

---

## 🔥 핵심 원인 한 줄

```txt
"조회 API에 side effect(조회수 증가)가 포함된 설계 문제"
+ "Next.js prefetch 동작이 트리거"
```

---

# ✅ 해결 방법 (우선순위 포함)

## 🥇 1순위 (베스트) → API 역할 분리

```txt
조회 API ≠ 조회수 증가
```

### 구조

```txt
GET  /posts/{id}        → 데이터 조회만
POST /posts/{id}/view   → 조회수 증가
```

### 👍 장점

- Next.js prefetch와 충돌 없음
- API 역할 명확 (REST 설계적으로도 맞음)
- SEO 유지
- UX 유지 (prefetch 그대로 사용 가능)

👉 **실무적으로 가장 올바른 해결 방법**

---

## 🥈 2순위 → prefetch 비활성화

```tsx
<Link href="/posts/1" prefetch={false} />
```

### 👍 장점

- 빠르게 문제 해결 가능

### 👎 단점

- 페이지 이동 시 느려짐 (prefetch 장점 상실)
- 일부 UX 저하

### ❗ SEO 영향

```txt
거의 없음
```

👉 **핫픽스용으로 적절**

---

## 🥉 3순위 → 클라이언트에서 API 호출

```txt
서버 컴포넌트 → 제거
클라이언트 useEffect에서 fetch
```

### 👍 장점

- prefetch 영향 완전히 제거

### 👎 단점 (중요)

- SEO 깨짐 ❌
- 초기 로딩 느려짐 ❌
- Next.js App Router 장점 포기 ❌

👉 **비추천 (특수한 경우 제외)**

---

# ❗ 추가로 꼭 짚고 넘어가야 할 부분

## 👉 generateMetadata 문제

너 코드에서 진짜 위험한 부분:

```ts
generateMetadata → fetchPostDetail 호출
```

이건:

```txt
prefetch 시점에도 실행됨
```

👉 즉:

```txt
"메타데이터 생성 = 조회수 증가 트리거"
```

---

## ✔ 해결 방향

### 방법 A (추천)

```txt
메타 전용 API 분리
```

### 방법 B

```txt
generateMetadata에서 fetch 제거
```

# 🔥 최종 결론

## ✔ 가장 좋은 구조

```txt
1. 조회 API는 순수하게 유지 (side effect 없음)
2. 조회수 증가 API 별도 분리
3. generateMetadata는 가벼운 API만 사용
4. prefetch는 유지 (Next.js 장점 살림)
```
